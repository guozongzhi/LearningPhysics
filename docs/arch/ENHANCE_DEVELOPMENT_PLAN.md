# LearningPhysics Enhance 开发计划

## 一、目标

在当前测验平台基础上，增加“主题录入 + 在线笔记 + 多人协作共创”能力，使学生和教师可以围绕物理知识主题共同创建、编辑、整理和沉淀学习内容。

## 二、产品方向

当前项目中的 `knowledge_nodes` 更适合作为知识点分类层，用于测验、统计和学习分析，不建议直接作为可编辑内容实体使用。

新增能力建议采用“文档层”设计，即在现有知识点之上增加可协作的主题文档系统，并将文档与一个或多个知识点关联。

核心目标：

- 支持围绕现有物理知识点创建主题笔记
- 支持多人协作，具备拥有者、编辑者、查看者角色
- 支持内容版本历史与回滚
- 支持将笔记与现有题库、测验流程联动
- 第一阶段优先追求稳定可用，不直接上实时协同编辑

## 三、范围建议

### 阶段 1：MVP

先交付一个可用的在线笔记版本，不引入实时协同复杂度。

功能范围：

- 创建、编辑、删除、浏览主题文档
- 文档可绑定一个或多个 `knowledge_nodes`
- 支持 Markdown 内容录入，并兼容 LaTeX 公式渲染
- 支持协作者管理
- 每次保存自动生成版本快照
- 支持按标题和知识点进行基础检索

### 阶段 2：教学流程增强

让主题文档真正进入学习闭环，而不是变成独立的笔记工具。

功能范围：

- 在文档中插入题目卡片与解析引用
- 支持评论或段落级批注
- 支持概念讲解、公式推导、实验记录等模板
- 支持班级或小组可见范围控制
- 支持文档活动记录

### 阶段 3：高级协作

在核心业务稳定后，再进一步提升多人编辑体验。

功能范围：

- 基于 WebSocket + CRDT 的实时协同
- 在线用户状态与编辑光标显示
- 并发编辑冲突辅助处理
- AI 摘要、提纲生成等增强能力

## 四、技术架构建议

### 后端

保持现有 `FastAPI + SQLModel` 结构不变，新增与 `topics`、`quiz`、`admin` 并行的文档模块。

建议新增模型：

- `TopicDocument`
  - `id`
  - `title`
  - `summary`
  - `content_markdown`
  - `owner_id`
  - `visibility`
  - `is_archived`
  - `created_at`
  - `updated_at`
- `TopicDocumentNode`
  - 文档与 `knowledge_nodes` 的关联表
- `TopicDocumentCollaborator`
  - `document_id`
  - `user_id`
  - `role`
- `TopicDocumentVersion`
  - `document_id`
  - `version_no`
  - `content_markdown`
  - `edited_by`
  - `created_at`
- `TopicDocumentComment`
  - 可放在第二阶段实现

建议新增 API：

- `GET /api/v1/documents`
- `POST /api/v1/documents`
- `GET /api/v1/documents/{id}`
- `PUT /api/v1/documents/{id}`
- `DELETE /api/v1/documents/{id}`
- `GET /api/v1/documents/{id}/versions`
- `POST /api/v1/documents/{id}/collaborators`
- `DELETE /api/v1/documents/{id}/collaborators/{user_id}`
- `GET /api/v1/knowledge-nodes/{id}/documents`

### 前端

保留现有测验主流程，不要把在线笔记直接混入当前首页的主题选题逻辑中，而是新增一个独立入口。

建议页面：

- `/notes`
- `/notes/new`
- `/notes/[id]`
- 可选后台管理入口 `/admin`

建议文档详情页布局：

- 顶部区域：标题、保存状态、协作者、权限信息
- 左侧区域：关联知识点、相关题目
- 中间区域：Markdown 编辑与预览
- 右侧区域：版本历史或评论区

## 五、实施计划

### 迭代 1：数据与接口基础

任务：

- 增加文档、协作者、知识点绑定、版本记录等表结构
- 增加对应的 Pydantic/Schema 定义
- 增加文档 CRUD 接口
- 增加 owner/editor/viewer 权限校验
- 增加创建、编辑、读取、权限控制测试

完成标准：

- 登录用户可以创建和编辑自己的文档
- 被授权协作者可以按角色进行编辑或查看
- 每次保存都会自动生成版本快照

### 迭代 2：前端 MVP

任务：

- 增加笔记列表页
- 增加笔记编辑页
- 接入 Markdown + LaTeX 渲染
- 增加知识点绑定与协作者管理界面
- 增加版本历史侧栏

完成标准：

- 用户可以通过界面创建主题文档
- 用户可以浏览、打开、编辑笔记
- 用户可以邀请协作者并完成保存

### 迭代 3：内容与学习联动

任务：

- 将文档与当前主题、题库数据打通
- 根据关联知识点展示相关题目
- 支持在笔记中插入题目引用
- 增加“基于当前笔记生成测验”入口

完成标准：

- 笔记功能不再是独立工具，而是进入现有学习闭环

### 迭代 4：评审与协作增强

任务：

- 增加评论功能
- 增加活动记录或通知
- 增加并发编辑冲突处理
- 评估是否引入实时协同技术栈

完成标准：

- 用户可以围绕同一篇主题笔记进行评审、讨论和低冲突协作

## 六、风险与取舍

- 如果直接复用 `knowledge_nodes` 作为内容实体，会把分类层和创作层耦合得过紧
- 实时协同编辑复杂度较高，不适合第一阶段直接投入
- 富文本编辑器在公式、版本控制和内容结构化方面维护成本更高，第一版更建议 Markdown
- 权限模型如果前期不设计清楚，后期共享和协作会非常难收敛

## 七、建议的下一步

建议先在增强版项目里启动“迭代 1”：

1. 新增文档相关数据模型
2. 暴露 CRUD 接口
3. 搭建最小可用的 `/notes` 和 `/notes/[id]` 页面
4. 先打通创建、编辑、读取流程，再继续扩展评论和实时协同
